SUMMARY
--------------------------------------------------------------------------------

This hard-difficulty task requires agents to debug conservation violations in a
C++ finite-volume adaptive mesh refinement (AMR) simulation. Six source modules
contain bugs that cause numerical drift through grid refinement, coarsening,
flux accumulation, serialization, and halo-exchange operations. The oracle
solution replaces all six buggy source files with corrected implementations,
rebuilds the project via CMake, and re-runs the test matrix. The test suite
contains 13 pytest functions that independently recompute reference values and
validate conservation, adjacency, moment invariance, snapshot integrity, and
anti-cheating markers.

--------------------------------------------------------------------------------
1. Test Dependencies Installed in Dockerfile Instead of test.sh
--------------------------------------------------------------------------------

File:    tbench-task/environment/Dockerfile (lines 21-23)
Problem: pytest==8.4.1 and pytest-json-ctrf==0.3.5 are installed in the Docker
         image. The test.sh script relies on these pre-installed packages rather
         than self-installing them, deviating from the standard test runner
         format that installs uv and pytest within test.sh itself.

Current code:
┌─────────────────────────────────────────────────────────────────────────────┐
│  RUN python -m pip install --no-cache-dir \                                 │
│      pytest==8.4.1 \                                                        │
│      pytest-json-ctrf==0.3.5                                                │
└─────────────────────────────────────────────────────────────────────────────┘

Suggested fix: Move pytest installation into tests/test.sh using the standard
uv-based pattern, and remove it from the Dockerfile:
┌─────────────────────────────────────────────────────────────────────────────┐
│  #!/bin/bash                                                                │
│  apt-get update && apt-get install -y curl                                  │
│  curl -LsSf https://astral.sh/uv/0.7.13/install.sh | sh                    │
│  source $HOME/.local/bin/env                                                │
│                                                                             │
│  if [ "$PWD" = "/" ]; then                                                  │
│    echo "Error: No working directory set."                                  │
│    exit 1                                                                   │
│  fi                                                                         │
│                                                                             │
│  uv venv .tbench-testing                                                    │
│  source .tbench-testing/bin/activate                                        │
│  uv pip install pytest==8.4.1 pytest-json-ctrf==0.3.5                       │
│                                                                             │
│  uv run pytest --ctrf /logs/verifier/ctrf.json /tests/test_outputs.py -rA   │
│                                                                             │
│  if [ $? -eq 0 ]; then                                                      │
│    echo 1 > /logs/verifier/reward.txt                                       │
│  else                                                                       │
│    echo 0 > /logs/verifier/reward.txt                                       │
│  fi                                                                         │
└─────────────────────────────────────────────────────────────────────────────┘

Explanation: Test-only dependencies should be installed in test.sh so the Docker
image remains lean and the test runner is self-contained. An exception exists if
the agent could reasonably use the dependency during solving — pytest arguably
qualifies, making this a warning rather than a critical issue.

--------------------------------------------------------------------------------
2. test.sh Missing Shebang Line
--------------------------------------------------------------------------------

File:    tbench-task/tests/test.sh (line 1)
Problem: The test.sh script uses `#!/bin/bash` which is correct. However, it
         does not install any dependencies itself (curl, uv, pytest), relying
         entirely on the Dockerfile. If the harness ever provides a minimal
         base environment, this test runner would fail.

Current approach: test.sh directly invokes `python -m pytest` without any
dependency installation step.

Suggested fix: Add the standard dependency installation preamble (curl, uv,
pytest) as shown in the fix for Warning #1.

Explanation: A self-contained test.sh is more robust across different harness
configurations and follows the standard Terminal-Bench pattern.

================================================================================
                            OVERALL ASSESSMENT
================================================================================

This is an excellent, well-crafted hard debugging task that authentically tests
deep systems reasoning across C++ memory layout, numerical conservation laws,
serialization, and adaptive mesh refinement. The only meaningful issue is the
non-standard test runner that installs dependencies via the Dockerfile.

Key Strengths:
  ✓ Genuinely challenging multi-module debugging requiring deep code
    comprehension across 6 interrelated C++ source files
  ✓ Independent reference simulation in test_outputs.py provides robust,
    non-trivial validation that cannot be trivially gamed
  ✓ Strong anti-cheating measures (canary marker check, fingerprint
    validation, independent recomputation rather than golden-file comparison)
  ✓ Complete and accurate documentation (layout.md, operations.md, cases.toml)
    gives agents the information they need without giving away the solution

Key Weaknesses:
  ✗ Test dependencies in Dockerfile rather than self-contained test.sh

Evaluates: C++ debugging, numerical computing (conservation laws), adaptive
           mesh refinement logic, binary serialization/deserialization
