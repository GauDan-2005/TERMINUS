The project in `/app` is a local plugin-style runtime with Node.js orchestration and a Rust helper invoked through `cargo run`. Repeated local runs should behave like one uninterrupted session: carried state stays consistent, reloads do not lose pending work, dependency changes stay isolated to the records that use them, and older records remain readable. Right now the report drifts depending on the run sequence.

Run `npm run matrix` from `/app`. The scenario runner writes `/app/outcome/report.json` with per-scenario records and a folded aggregate. Each record carries its scenario name, ordered rows, a closing finalCounter, dependencyTotal, acceptedOlder, and healthy flag. Each row exposes index, value, factor, nativeValue, and a sorted deferred list. The aggregate reports source, recordCount, totalCounter, dependencyTotal, rowTotal, healthyCount, and status.

Each cycle's native row feeds the next cycle, and a scenario's closing counter should reflect the last native row produced—not merely the last input base. Older native revision markers must normalize like newer ones, incorporating carried state into the native row. Deferred numbers listed on a cycle belong to that same cycle's row, not the next. Descriptor factors are recomputed per row from that row's descriptor number and position, without cross-row caching. A scenario is healthy when older records are accepted and its closing counter equals the sum of input bases. Aggregate status reads healthy when every scenario is healthy, otherwise stale. Counters, dependency totals, row totals, and status must be derived from the records, not separate fixtures.

Repair bugs in the existing Node modules and Rust crate. Do not replace the runner with hardcoded JSON or a single-script reimplementation that skips cargo run. Invoking cargo run compiles the native helper under `/app/target`.
