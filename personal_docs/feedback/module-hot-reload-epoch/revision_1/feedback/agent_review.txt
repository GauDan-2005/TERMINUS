SUMMARY
--------------------------------------------------------------------------------

This task requires debugging a plugin-style runtime with Node.js orchestration
and a Rust native helper. The agent must identify and fix logic bugs across
five source files (four JS modules and one Rust crate) so that `npm run matrix`
produces correct per-scenario records and a folded aggregate in
`/app/outcome/report.json`. The solution repairs carry-state propagation,
deferred-value ownership, descriptor-factor caching, aggregate health
computation, and native-revision normalization. The test suite contains 12
functions that verify correct output for six scenarios, aggregate integrity,
Rust compilation, and deterministic re-runs.

1. Test Dependencies Installed in Dockerfile
--------------------------------------------------------------------------------

File:    tbench-task/environment/Dockerfile (lines 24-26)
Problem: pytest and pytest-json-ctrf are installed in the Docker image rather
         than in test.sh. While this is documented as intentional (test.sh
         comment at line 3-4) and isolated in /opt/pytest venv, it deviates
         from the standard pattern where test.sh installs its own dependencies.

Current code:
┌─────────────────────────────────────────────────────────────────────────────┐
│  RUN /opt/pytest/bin/pip install --no-cache-dir \                           │
│          pytest==8.4.1 \                                                    │
│          pytest-json-ctrf==0.3.5                                            │
└─────────────────────────────────────────────────────────────────────────────┘

Explanation: The venv isolation at /opt/pytest mitigates agent access concerns,
and the test.sh comment documents this as deliberate. However, agents can
still discover and invoke pytest from PATH. Acceptable if this is a
project-level convention, but worth noting for portability.
