# Submission Questions — overlayfs-whiteout-flatten

## 1. Difficulty Explanation

The task ships a Go userspace OCI image-layer flattener with four independent bugs, one in each stage of its merge pipeline: opaque-directory handling, per-name whiteout, attribute copy-up, and hard-link materialization. It's hard because no single fix repairs most of the failures, and the instruction is symptoms-only. The agent is told the flattened tree drifts from what a kernel overlay mount would show, but not which stage is wrong or how. Each buggy stage produces output that's internally consistent and looks correct on plain stacks, so the divergence only surfaces on stacks that actually use whiteouts, opaque markers, repeated paths, or hard links. The agent has to reproduce the drift, then cross-reference the produced tree against the documented overlay contract one behavior at a time.

The overlay rules themselves are subtle and easy to get almost right, which is where both humans and agents lose time. An opaque marker must mask only the lower layers' contributions while keeping the marking layer's own entries and any higher layer's additions. A per-name whiteout must delete the entire subtree under the name, not just the exact path. Copy-up must take the topmost layer's permissions and ownership, not a lower one's. Hard-linked members must resolve to a single shared inode rather than independent copies. The buggy code for each reads as plausible: "delete the whiteout target," for example, looks fine until you realize it has to cover the descendants. The package names are deliberately neutral (sift/cull/vouch/braid) so a symptom can't be grepped straight to a file, and decoy reporting helpers add more misdirection.

## 2. Solution Explanation

The approach is to build and run the tool on a bundled stack, diff the result against the rules in the environment docs, and localize the divergence to the post-accumulation passes. The layer index is accumulated correctly, so the faults are in how the four passes resolve and materialize it. The four symptoms are independent, and each maps to exactly one overlay rule and one stage, so the agent has to diagnose and repair all four instead of chasing a single root cause.

Each pass needs its own fix. The opaque pass must restrict a sealed directory so it suppresses only the contributions from levels below the marker, keeping the marking layer and the layers above it, instead of dropping everything under the directory. The whiteout pass must remove the named path along with its whole descendant range for the layers at or below the marker, not just the exact path. The copy-up pass must take the attribute record from the highest layer that provides a path, its Top, not its Lowest. The materialization pass must write the first member of a sharing set and hard-link the rest to it, instead of writing each as a separate file. The fixes are each small and local, but they have to coordinate: a whiteout that removes one hard-link member, for instance, still has to leave the survivors sharing one inode.

## 3. Verification Explanation

The verifier never trusts a golden tree or the tool's own report. Each test builds one or more layer tarballs in-process, with the ground truth known by construction, and runs the flattener on them. It then walks the produced directory tree and asserts the overlay-equivalent invariants directly: which entries survive and which are hidden, which layer's permission bits and ownership win, and which paths share one backing object (the same inode, with a link count of at least the number of surviving members). Because the scenarios are built at verify time, the agent can't satisfy the tests by hand-writing an output tree.

The cases are partitioned so each stage is exercised on its own: opaque scoping, subtree whiteout, topmost copy-up, and hard-link sharing each have dedicated tests. A whole-pipeline parity test over the bundled stacks plus seeded randomized stacks then confirms the stages compose correctly, and a re-flatten check enforces determinism. Ownership assertions run only as root, so the suite stays stable in non-root sandboxes. The reward reaches 1.0 only when all four stages are correct: any partial fix leaves a discriminating subset of tests failing. That's what keeps the difficulty honest.
