The project in `/app` is a local plugin-style toolchain: Node.js orchestration calls a small Rust helper through `cargo run`, and the two cooperate to fold a batch of scenarios into one summary. Right now the summary is unreliable. Repeating the same batch can produce a different file from one attempt to the next, some scenarios are reported with numbers that do not reconcile with their inputs, the folded totals disagree with the per-scenario figures they claim to sum, and the overall status reports stale even though nothing is actually wrong with the inputs. Treat these as symptoms of defects in the existing code, not as expected behavior.

Run `npm run matrix` from `/app`. The scenario runner reads the shipped inputs and writes `/app/outcome/report.json` holding per-scenario records and one folded aggregate. Each record carries its scenario name, its ordered rows, a closing finalCounter, a dependencyTotal, an acceptedOlder flag, and a healthy flag. Each row exposes its index, a value, a factor, a nativeValue, and a sorted deferred list. The aggregate carries a source, a recordCount, a totalCounter, a dependencyTotal, a rowTotal, a healthyCount, and a status that should read healthy when every record is healthy and stale otherwise.

Your job is to make the report correct and reproducible: every per-scenario figure must follow from that scenario's own inputs together with the helper's output, every aggregate figure must be folded from the records it summarizes, and two runs over unchanged inputs must agree. Study the runner, the helper, and the shipped inputs to work out what each field is supposed to mean, then repair the defects in the existing Node modules and the Rust crate in place so the numbers reconcile and the status reflects reality. Do not replace the runner with hardcoded JSON or collapse it into a single script that skips the `cargo run` step; invoking `cargo run` is what compiles the native helper under `/app/target`.
