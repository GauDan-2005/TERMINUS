The Rust workload runner under `/app` sometimes leaves its audit stalled or inconsistent with bounded queues, cancelled branches, and child work mixed. Fix it so `bash /app/scripts/run-matrix.sh` completes and writes `/app/output/executor-audit.json` for the shipped cases from the current inputs.

The JSON has a top-level cases array for amber, cobalt, drake, ember, flint, and graphite. Each case reports name, ok, finished, pending, trace, ledger, and journal. Trace is the accepted work records in acceptance order, where lower accepted turn comes first, lanes are ordered by name within a turn, and ready work within a lane keeps plan-file order. Every record carries case, task, parent, depth, lane, and turn; turn is when that record is accepted. A root record, one with no parent of its own, still carries parent as the literal string `"-"` (never null or empty) in the JSON trace, the ledger JSONL, and the journal. Ledger points to a JSONL file containing the same records in the same order. Journal points to a text file with whitespace-delimited rows in the same order: case task parent depth lane turn.

A clean audit has every case finished, has no pending records, has ok true, preserves parent/depth relationships for child work, keeps cancelled branches and their descendants out of the accepted trace, and leaves the JSON, ledger, and journal in agreement. Child rows keep parent and depth on accepted records for hierarchy only; they do not wait for the named parent to be accepted first. Each row becomes eligible on its own arrival turn, and CAP, lane order, and plan-file order decide acceptance from there. Removing a branch drops that name and every descendant linked through the child chain, not only rows with an identical label. A CAP row limits how many ready records each lane may accept in one turn; overflow remains live for later turns and is considered alongside work that becomes ready later. The single-case helper at `/app/scripts/run-one.sh` must obey the same contract for one shipped case as the matrix run does for all cases. Repeated logical child entries should appear once in the accepted trace, and two fresh matrix runs from unchanged inputs should parse to equivalent audit data.
