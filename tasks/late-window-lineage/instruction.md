The event engine under `/app` currently produces restart-sensitive audit output for late-window scenarios that is not clean; fix the implementation so running `bash /app/scripts/run-matrix.sh` writes a clean `/app/output/window-audit.json` from the current case inputs rather than a canned report.

The JSON object has a single cases array for aurora, boreal, cirrus, drift, ember, and flux, in that order, with no audit keys beyond the shape described here. Each case reports only name, ok, direct, replay, and repeat. Direct and replay each contain only totals and corrections for the same input run through uninterrupted and restarted paths. Totals list partition, window, key, and value fields, ordered by partition, window, then key; value is the final running accumulation for that partition, window, and key group. Corrections list partition, window, key, value, id, and lineage fields. Corrections are an append log, not only final summaries: each accepted unique input row in a partition, window, and key group contributes a correction row whose value is the group's running accumulation through that row and whose lineage contains that group's unique input ids through that row, ordered by ascending input-row sequence rather than file position. Input ids are case-wide; if duplicates appear, the accepted row is the one with the earliest input-row sequence, even when duplicate rows occur under different partitions, windows, keys, or file positions. Correction rows are emitted deterministically by partition, window, key, then lineage. Each correction id is exactly the case name, partition, window, key, value, and every lineage member in that order, separated by single colon characters with no extra fields or prefixes; when a group has multiple lineage members, append each member after value in ascending input-row sequence, again separated by colons. Input row ids are opaque strings and may contain pipe or colon characters themselves; any delimiter-separated rows written for restart or trace persistence must round-trip those characters without dropping, splitting, or corrupting fields. Repeat reports whether a second restarted run emitted the same corrections.

The audit is clean when every case has ok true, direct and replay totals match, correction ids and lineage stay stable across repeated restarted runs, and replayed input does not create duplicate correction records.
